# trade-ledger

> A canonical Postgres-backed trade lifecycle ledger for algorithmic trading systems. Mints trade IDs, writes one row per trade, and enforces P&L consistency at the library boundary.

[![CI](https://github.com/arieslao/trade-ledger/actions/workflows/ci.yml/badge.svg)](https://github.com/arieslao/trade-ledger/actions/workflows/ci.yml)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)

## What this solves

When you build a trading system across multiple strategies, you eventually accumulate **multiple sources of P&L truth** — the execution log, the scorecard, the broker reconciliation, the dashboard estimate. They drift. You end up with three different P&L numbers for the same trade and no way to know which one is right.

`trade-ledger` is a small library that fixes this at the library boundary:

- **One canonical `trades` Postgres table** — single source of truth for entry → exit → P&L.
- **One mint point for `trade_id`** — every strategy gets its ID from the same function, so cross-system joins always work.
- **P&L derived in exactly one place** — `pnl_dollars` and `pnl_pct` are always derived together from `(entry_price, exit_price, qty, asset_class, direction)`, so they can never disagree in sign or magnitude.
- **Mark-vs-fill mismatch detection** — if a caller passes a broker's mark-based `pnl_dollars` that disagrees in sign with the fill-derived value, the library rejects the override (and stashes it in metadata for audit) instead of silently storing inconsistent numbers.
- **Broker-acceptance gate** — refuses to write a `mode='live'` row when the broker accepted no leg, including the case of a 200-OK order with `status='rejected'` (which an id-presence check would miss).
- **Idempotent open/close** — duplicate `open_trade` is a no-op return; re-`close` is a no-op skip.
- **Never raises** — DB and external errors degrade to a fallback JSONL so trades are never silently dropped, while the calling engine keeps running.

## Quick start

```python
import os
os.environ["DATABASE_URL"] = "postgresql://user:pass@host/db"

from trade_ledger import open_trade, close_trade, find_open_trade

# 1) Open a live trade after broker acceptance
r = open_trade(
    method="momentum_breakout",
    asset_class="options",
    symbol="SPY",
    direction="bullish",
    entry_price=1.25,
    qty=2,
    contract={"symbol": "SPY250620C00450000", "strike": 450, "expiry": "2025-06-20", "right": "C"},
    stop=0.90,
    target=2.00,
    accepted=True,         # broker filled at least one leg
    mode="live",
)
trade_id = r["trade_id"]

# 2) Later, close it
close_trade(
    trade_id=trade_id,
    exit_price=2.10,
    reason="target_hit",
    exit_price_source="fill",
    pnl_source="broker",
)

# 3) From an exit monitor, look up the open trade for a position about to close
existing = find_open_trade(
    method="momentum_breakout",
    symbol="SPY",
    contract_occ="SPY250620C00450000",
)
```

See [`examples/basic_usage.py`](examples/basic_usage.py) for a fuller walkthrough.

## Schema

The canonical `trades` table — see [`migrations/001_create_trades.sql`](migrations/001_create_trades.sql). Key columns:

| Column | Notes |
|---|---|
| `trade_id` PK | Generated by `generate_trade_id(method, symbol)`. Format: `{method}_{symbol}_{ts}_{4hex}`. |
| `method`, `asset_class`, `symbol`, `direction`, `qty`, `contract` | Identity + sizing. `contract` is JSONB — supports single-leg + multi-leg spreads. |
| `entry_ts`, `entry_price`, `entry_price_source` | `entry_price_source ∈ {fill, mid, mark, estimate}`. |
| `exit_ts`, `exit_price`, `exit_price_source`, `exit_reason` | Filled at close. |
| `pnl_dollars`, `pnl_pct`, `pnl_source` | Derived together. `pnl_source ∈ {broker, computed, estimate}`. |
| `status` | `open` / `closed` / `partial` / `expired` / `orphaned`. |
| `mode` | `live` / `shadow` / `paper`. Shadow rows enable counterfactual analysis (see below). |
| `confidence_tier` | `native` (real fill data) / `reconstructed` (backfilled from logs) / `estimate` (synthetic). Lets dashboards distinguish trustworthy from best-effort rows. |
| `policy_version` | Versions the row's schema/semantics so migrations can be reasoned about. |
| `metadata` | JSONB free-form. Library stashes diagnostic info here (e.g. rejected supplied P&L). |

## Shadow trades (counterfactual analysis)

When your executor chooses between alternatives (e.g. simple call vs. vertical spread), `mode='shadow'` rows track the *unchosen* path so you can later measure executor selection bias:

```python
live = open_trade(method="overnight", mode="live", contract=chosen, ...)
shadow = open_trade(
    method="overnight",
    mode="shadow",
    contract=unchosen,
    entry_price=unchosen_mid,
    price_source="mark",
    metadata={"shadow_pair_of": live["trade_id"], "shadow_reason": "executor_unselected"},
)
```

A single SQL join then answers "is my executor picking right?":

```sql
SELECT
  live.method,
  AVG(live.pnl_pct)                    AS live_avg_return,
  AVG(shadow.pnl_pct)                  AS shadow_avg_return,
  AVG(shadow.pnl_pct - live.pnl_pct)   AS shadow_minus_live_edge,
  COUNT(*)                             AS pairs
FROM trades live
JOIN trades shadow
  ON shadow.metadata->>'shadow_pair_of' = live.trade_id
WHERE live.mode = 'live' AND shadow.mode = 'shadow'
  AND live.status IN ('closed','expired')
GROUP BY live.method;
```

If `shadow_minus_live_edge > 0` for a method, the executor is systematically picking the worse alternative.

## Configuration

| Env var | Purpose | Default |
|---|---|---|
| `DATABASE_URL` | Postgres connection string. **Required** for any DB op. | (unset) |
| `TRADE_LEDGER_AUDIT_LOG` | Append-only JSONL audit log of every open/close. | `./trade_ledger.jsonl` |
| `TRADE_LEDGER_FALLBACK_LOG` | JSONL written when a DB write fails (so trades are never lost). | `./trade_ledger_errors.jsonl` |

## Notification hooks (Discord, Slack, ntfy, etc.)

`trade-ledger` does not depend on any specific notifier. Register your own:

```python
from trade_ledger import register_entry_hook, register_exit_hook

def my_discord_entry(*, trade_id, method, symbol, direction, entry, stop, target, **kwargs):
    # Post to Discord, return {"message_id": "...", "posted": True}
    ...

def my_discord_exit(*, trade_id, symbol, method, exit_price, reason, pnl_pct, pnl_dollars,
                    reply_to_message_id, **kwargs):
    ...

register_entry_hook(my_discord_entry)
register_exit_hook(my_discord_exit)
```

Hooks are best-effort: if they raise, the trade row is still written.

## Design principles

1. **Errors never propagate up.** The trade row is the most important artifact; the comms post and the audit log are best-effort. Errors return in the result dict and (on DB failure) get persisted to a fallback JSONL so a reconciler can replay them.
2. **One place to derive P&L.** `compute_pnl()` is the only function in your codebase that should produce `pnl_dollars` and `pnl_pct`. If something else computes P&L, you'll eventually have inconsistent numbers in the same row.
3. **Mode is a first-class field.** Distinguishing `live`, `shadow`, and `paper` rows in the schema (not in client code) makes counterfactual analysis and the executor-selection question above tractable with a single SQL join.
4. **Long-premium options are not direction-flipped.** A bearish PUT held long is still a winner when its premium rises. Direction sign only applies to assets you can actually short (equity, forex, crypto).
5. **Confidence tier is honest.** Pre-standard rows reconstructed from logs are tagged `reconstructed`, not pretended to be `native`. Dashboards can show full history while flagging which rows are best-effort.
